# ТЕХНИЧЕСКОЕ ЗАДАНИЕ v2.0
## Торговый бот для Gate.io с автоматической торговлей новыми фьючерсными листингами

---

## 1. ОБЩИЕ ПОЛОЖЕНИЯ

**Проект:** Разработка торгового бота для автоматической торговли новыми фьючерсными контрактами USDT на бирже Gate.io

**Стоимость:**
- Основной бот: 35 000 руб.
- Telegram-бот: 5 000 руб.
- **Итого:** 40 000 руб.

**Срок разработки:** 3 недели (120 часов)

**Технологический стек:**
- Python 3.10+
- Официальный Gate.io Python SDK
- asyncio, aiohttp/websockets
- PostgreSQL (для хранения истории контрактов, позиций, усреднений)
- Redis (для кэширования и быстрых операций)
- Telegram Bot API (aiogram 3.x)
- YAML/.env для конфигурации
- Sentry для мониторинга ошибок

## 2. ОСНОВНАЯ СТРАТЕГИЯ

### 2.1. Направление торговли
- **Только SHORT** (шорт) по новым фьючерсным листингам

### 2.2. Логика входа в позицию

**Первичные условия (новый листинг):**
1. Обнаружен новый фьючерсный контракт (отсутствует в локальной БД)
2. Поле `launch_time` из API указывает на новый контракт
3. Контракт записывается в БД с меткой `listing_taken_in_work = true`
4. Стакан заполнен маркетмейкером:
   - Минимум 5 уровней с обеих сторон bid/ask
   - Достаточный объем на лучшем уровне (настраиваемый порог в USDT)
   - Спред не превышает заданный процент
5. Ожидание заполнения стакана: до 10 минут

**Вторичные условия (повторные входы после фиксации прибыли):**
1. `ATH ratio > 0.3` (текущая цена / ATH)
2. С момента листинга прошло менее 30 дней (настраиваемый параметр)
3. Монета уже помечена как `listing_taken_in_work`

**Расчет ATH ratio:**
```
ATH ratio = Текущая цена / Максимальная историческая цена (ATH)
```
- ATH определяется по недельным свечам через Gate.io API
- Без использования сторонних сервисов (CoinGecko и т.п. не используются)

### 2.3. Фиксация прибыли

**Основная логика:**
- При движении цены вниз на **2%** (настраиваемый параметр) фиксировать прибыль
- Закрытие позиции рыночным ордером
- Сразу переоткрыть шорт тем же объемом
- Повторять цикл пока выполняются условия:
  - `ATH ratio > 0.3`
  - `< 30 дней` с момента листинга (или другого настроенного периода)

**Расчет процента движения ("линейка" как на Coinglass):**
```
Процент роста = ((Текущая цена - Цена входа) / Цена входа) × 100%
```
- Для шорта: отрицательный процент = прибыль
- Фиксация при достижении -2% (или другого настроенного значения)

**Реализация:**
- Закрытие и переоткрытие: двумя последовательными рыночными ордерами
- При быстром пролете уровня: закрытие по рынку без ожидания отката

### 2.4. Сетка усреднений

**Особенность для Gate.io:**
- **НЕ жёсткое усреднение "количество монет"**
- **Фиксированная сумма в USDT** для каждого усреднения (настраиваемая)
- Процент роста считается от **первоначальной цены входа**

**Уровни усреднения от цены входа (по умолчанию):**
- 300%
- 700%
- 1000%

**Правила усреднения:**
1. Объем каждого усреднения = фиксированной сумме в USDT (настраиваемой через ТГ)
2. Объем может отличаться от начального объема позиции
3. Максимум N усреднений (по умолчанию 3, настраивается через ТГ)
4. Уровни усреднений настраиваются через ТГ (например: 300%, 700%, 1000%)
5. После усреднения 2% профит считается от средней цены входа

**Пример работы:**
```
Начальная позиция: шорт $100, объем 10 USDT
Цена выросла на 300% → $400
Усреднение 1: открываем дополнительный шорт на 10 USDT (настраиваемая сумма)
Цена выросла на 700% от начальной → $800
Усреднение 2: открываем дополнительный шорт на 10 USDT
Цена выросла на 1000% от начальной → $1100
Усреднение 3: открываем дополнительный шорт на 10 USDT
```

**Важное требование - история усреднений:**
- База должна хранить историю всех усреднений для каждой монеты:
  - Количество усреднений
  - Уровни роста, на которых были усреднения
  - Суммы усреднений
- При изменении настроек суммы/сетки усреднений через ТГ:
  - Учитывать уже произошедшие усреднения
  - Не повторять усреднения на уже пройденных уровнях
  - Продолжать работу с учетом новой конфигурации

**Пример сценария:**
```
Монета была усреднена на 300% по 10 USDT
Цена сейчас на 700% от первой точки входа
В ТГ меняем сумму усреднения на 15 USDT
Далее усреднение на 700% будет на 15 USDT (с учетом, что на 300% уже было)
```

**Возможность несимметричных усреднений:**
- Для новых листингов с высокой волатильностью можно задать:
  - Уровни: [300%, 700%, 1000%]
  - Суммы: [10 USDT, 10 USDT, 10 USDT] или разные суммы

### 3.1. Модуль мониторинга новых листингов ⏱ 12 часов

**Функции:**
- Периодический REST-поллинг `/api/v4/futures/usdt/contracts` каждые 5 секунд
- Сверка с локальной базой данных PostgreSQL
- Использование поля `launch_time` для подтверждения новизны
- Запись новых контрактов в БД с полями:
  ```sql

  - launch_time
  - status (new/listing_taken_in_work/completed)
  - first_seen_at
  - ath_price (заполняется отдельно)
  - ath_updated_at
  ```

**Защита от ложных срабатываний:**
- Ведение полной истории всех когда-либо виденных пар
- Проверка по `launch_time` и наличию в БД
- Исключение повторной торговли по тем же контрактам
- Отсечение возобновления торгов от реальных листингов

**API лимиты:**
- Gate позволяет 200 запросов / 10 сек на endpoint
- Используем ~2 запроса за 10 сек → запас 100x

### 3.2. Модуль анализа стакана (WebSocket) ⏱ 16 часов

**Функции:**
- Подключение к `wss://fx-ws.gateio.ws/v4/ws/usdt` при обнаружении нового тикера
- Подписка на канал `order_book` для конкретного контракта
- Мониторинг в реальном времени:
  - Глубина стакана (bid/ask)
  - Объемы на каждом уровне
  - Спред между лучшими ценами
  - Скорость обновлений стакана
  - Баланс давления покупателей/продавцов (bid volume / ask volume)

**Условия готовности стакана:**
- Минимум 5 уровней с обеих сторон (настраиваемо)
- Достаточный объем на лучшем уровне (настраиваемый параметр в USDT)

- Статус: "MM ACTIVE" - маркетмейкер заполнил стакан

**Ожидание заполнения ММ:**
- До 10 минут после обнаружения листинга (настраиваемо)
- Таймаут: если стакан не заполнен → пропуск листинга
- Вход только при выполнении всех условий

**WebSocket без лимитов** - публичный канал Gate.io

### 3.3. Торговый модуль ⏱ 20 часов

**Функции:**
- Расчет размера позиции на основе настроек из ТГ
- Размещение рыночных ордеров (открытие/закрытие)
- Управление открытыми позициями
- Реализация сетки усреднений с учетом истории
- Фиксация прибыли с переоткрытием позиции
- Расчет средней цены входа после усреднений

**Параметры позиции:**
- Начальный объем: настраивается через ТГ (по умолчанию 10 USDT)
- Для тестов: минимально возможный объем на Gate с максимальным плечом
- Плечо: максимальное (для выживания при росте до 1000%)
- Стоп-лосс: **НЕ ИСПОЛЬЗУЕТСЯ** (стратегия бесконечного шорта)

**Логика усреднений:**
```python
# Псевдокод логики
if not position_in_db:
    # Первое усреднение
    avg_count = 0
else:
    avg_count = position.avg_count  # из БД
    avg_levels = position.avg_levels  # [300, 700] уже было

current_growth_pct = calculate_growth_from_entry(entry_price, current_price)

# Проверяем каждый настроенный уровень
for level in configured_levels:
    if level not in avg_levels and current_growth_pct >= level:
        if avg_count < max_avg_count:
            open_additional_short(avg_amount_usdt)
            save_avg_to_db(level, avg_amount_usdt)
```

**Фиксация прибыли:**
```python
# Псевдокод
unrealized_pnl_pct = calculate_pnl_pct(entry_price, current_price)

if unrealized_pnl_pct <= -take_profit_pct:  # -2% для шорта
    close_position(market_order)
    if can_reopen(ath_ratio, days_since_listing):
        reopen_position(initial_volume_usdt)
    mark_listing_completed_if_needed()
```

### 3.8. Модуль базы данных ⏱ 6 часов

**Схема PostgreSQL:**

**Таблица contracts:**
```sql
CREATE TABLE contracts (
    id SERIAL PRIMARY KEY,
    symbol VARCHAR(50) UNIQUE NOT NULL,
    launch_time TIMESTAMP NOT NULL,
    status VARCHAR(20) DEFAULT 'new',  -- new, listing_taken_in_work, completed
    first_seen_at TIMESTAMP DEFAULT NOW(),
    ath_price DECIMAL(20, 8),
    ath_updated_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_contracts_symbol ON contracts(symbol);
CREATE INDEX idx_contracts_status ON contracts(status);
```

**Таблица positions:**
```sql
CREATE TABLE positions (
    id SERIAL PRIMARY KEY,
    contract_symbol VARCHAR(50) NOT NULL,
    entry_price DECIMAL(20, 8) NOT NULL,
    current_price DECIMAL(20, 8),
    volume_usdt DECIMAL(20, 8) NOT NULL,
    avg_count INTEGER DEFAULT 0,
    unrealized_pnl DECIMAL(20, 8),
    status VARCHAR(20) DEFAULT 'open',  -- open, closed
    days_since_listing INTEGER,
    opened_at TIMESTAMP DEFAULT NOW(),
    closed_at TIMESTAMP,
    updated_at TIMESTAMP DEFAULT NOW()
);
```

**Таблица averaging_history:**
```sql
CREATE TABLE averaging_history (
    id SERIAL PRIMARY KEY,
    position_id INTEGER REFERENCES positions(id),
    contract_symbol VARCHAR(50) NOT NULL,
    avg_level_pct DECIMAL(10, 2) NOT NULL,  -- 300, 700, 1000
    avg_amount_usdt DECIMAL(20, 8) NOT NULL,
    avg_price DECIMAL(20, 8) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);
```

**Таблица trades:**
```sql
CREATE TABLE trades (
    id SERIAL PRIMARY KEY,
    contract_symbol VARCHAR(50) NOT NULL,
    trade_type VARCHAR(20) NOT NULL,  -- open, close, avg_open, avg_close
    price DECIMAL(20, 8) NOT NULL,
    volume_usdt DECIMAL(20, 8) NOT NULL,
    pnl DECIMAL(20, 8),
    fee DECIMAL(20, 8),
    created_at TIMESTAMP DEFAULT NOW()
);
```

**Таблица settings:**
```sql
CREATE TABLE settings (
    id SERIAL PRIMARY KEY,
    param_name VARCHAR(50) UNIQUE NOT NULL,
    param_value TEXT NOT NULL,
    param_type VARCHAR(10),  -- int, float, str, json
    updated_at TIMESTAMP DEFAULT NOW()
);
```

## 5. API ЛИМИТЫ GATE.IO ⏱ 2 часа (интеграция)

**Публичные endpoints (мониторинг):**
- `/api/v4/futures/usdt/contracts`: **200 запросов / 10 сек**
- Используем ~2 запроса за 10 сек → **запас 100x**

**Приватные endpoints (торговля):**
- Размещение ордеров: **100 запросов/сек**
- Отмена ордеров: **200 запросов/сек**
- Остальные приватные: **200 запросов / 10 сек**

**WebSocket:** без лимитов (публичный канал)

**Fill ratio правило:**
- Если за 24ч более 86,400 запросов и ни один ордер не исполнен → 1 запрос/сек
- Если fill ratio менее 1% → 2 запрос/сек
- Для нашей стратегии не проблема (ордера исполняются)

**Вывод:** лимитов хватит с большим запасом
