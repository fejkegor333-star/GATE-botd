# ТЕХНИЧЕСКОЕ ЗАДАНИЕ
## Автоматический торговый бот для фьючерсных листингов Gate.io

**Версия:** 2.0
**Дата:** 14.03.2026
**Статус:** Утверждено

---

## 1. ЦЕЛЬ ПРОЕКТА

Разработка автоматического торгового бота для торговли новыми фьючерсными контрактами USDT на бирже Gate.io. Стратегия основана на открытии коротких позиций (SHORT) по новым листингам с последующей фиксацией прибыли и сеткой усреднений при росте цены.

---

## 2. ОБЩИЕ ТРЕБОВАНИЯ

### 2.1. Технологический стек

- **Язык программирования:** Python 3.10+
- **API интеграция:** Официальный Gate.io Python SDK
- **Асинхронность:** asyncio, aiohttp
- **WebSocket:** websockets или aiohttp
- **База данных:** PostgreSQL 14+
- **Кэширование:** Redis (опционально)
- **Конфигурация:** YAML / .env файлы
- **Логирование:** Python logging + Sentry
- **Telegram бот:** aiogram 3.x

### 2.2. Требования к коду

- Type hints для всех функций
- Docstrings в формате Google или NumPy
- PEP 8 стандарт
- Модульная архитектура
- Обработка ошибок и исключений
- Логирование всех критических операций

### 2.3. Требования к безопасности

- API ключи хранятся в .env файле
- Минимальные права API ключей
- Проверка баланса перед каждой операцией
- Circuit Breaker при критических ошибках
- Логирование всех торговых операций

### 3.2. База данных

**Схема PostgreSQL:**

#### Таблица `contracts`
Хранит информацию о всех фьючерсных контрактах

```sql
CREATE TABLE contracts (
    id SERIAL PRIMARY KEY,
    symbol VARCHAR(50) UNIQUE NOT NULL,
    launch_time TIMESTAMP NOT NULL,
    status VARCHAR(20) DEFAULT 'new',  -- new, in_work, completed
    first_seen_at TIMESTAMP DEFAULT NOW(),
    ath_price DECIMAL(20, 8),
    ath_updated_at TIMESTAMP,
    listing_taken_in_work BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_contracts_symbol ON contracts(symbol);
CREATE INDEX idx_contracts_status ON contracts(status);
CREATE INDEX idx_contracts_launch_time ON contracts(launch_time);
```

#### Таблица `positions`
Хранит информацию об открытых и закрытых позициях

```sql
CREATE TABLE positions (
    id SERIAL PRIMARY KEY,
    contract_symbol VARCHAR(50) NOT NULL REFERENCES contracts(symbol),
    entry_price DECIMAL(20, 8) NOT NULL,
    current_price DECIMAL(20, 8),
    total_volume_usdt DECIMAL(20, 8) NOT NULL,
    avg_count INTEGER DEFAULT 0,
    unrealized_pnl DECIMAL(20, 8),
    status VARCHAR(20) DEFAULT 'open',  -- open, closed
    days_since_listing INTEGER DEFAULT 0,
    opened_at TIMESTAMP DEFAULT NOW(),
    closed_at TIMESTAMP,
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_positions_symbol ON positions(contract_symbol);
CREATE INDEX idx_positions_status ON positions(status);
```

#### Таблица `averaging_history`
История усреднений по каждой позиции

```sql
CREATE TABLE averaging_history (
    id SERIAL PRIMARY KEY,
    position_id INTEGER NOT NULL REFERENCES positions(id),
    contract_symbol VARCHAR(50) NOT NULL,
    avg_number INTEGER NOT NULL,  -- 1, 2, 3...
    avg_level_pct DECIMAL(10, 2) NOT NULL,  -- 300, 700, 1000
    avg_amount_usdt DECIMAL(20, 8) NOT NULL,
    avg_price DECIMAL(20, 8) NOT NULL,
    avg_entry_price DECIMAL(20, 8) NOT NULL,  -- Средняя цена после усреднения
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_avg_history_position ON averaging_history(position_id);
CREATE INDEX idx_avg_history_symbol ON averaging_history(contract_symbol);
```

#### Таблица `trades`
История всех торговых операций

```sql
CREATE TABLE trades (
    id SERIAL PRIMARY KEY,
    contract_symbol VARCHAR(50) NOT NULL,
    trade_type VARCHAR(20) NOT NULL,  -- open, close, avg_open, tp_close
    price DECIMAL(20, 8) NOT NULL,
    volume_usdt DECIMAL(20, 8) NOT NULL,
    pnl DECIMAL(20, 8),
    fee DECIMAL(20, 8),
    order_id VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_trades_symbol ON trades(contract_symbol);
CREATE INDEX idx_trades_type ON trades(trade_type);
CREATE INDEX idx_trades_created ON trades(created_at);
```

#### Таблица `settings`
Хранение настроек бота

```sql
CREATE TABLE settings (
    id SERIAL PRIMARY KEY,
    param_name VARCHAR(50) UNIQUE NOT NULL,
    param_value TEXT NOT NULL,
    param_type VARCHAR(10) NOT NULL,  -- int, float, str, json, bool
    description TEXT,
    updated_at TIMESTAMP DEFAULT NOW(),
    updated_by VARCHAR(50) DEFAULT 'system'  -- system, telegram
);

CREATE INDEX idx_settings_name ON settings(param_name);
```

## 4. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### 4.1. Модуль мониторинга новых листингов

**Назначение:** Обнаружение новых фьючерсных контрактов на Gate.io

#### 4.1.1. Функции

**F1.1. Периодический опрос контрактов**

- Выполнять REST запрос к `/api/v4/futures/usdt/contracts` каждые 5 секунд
- Получать список всех контрактов с полями: `name`, `symbol`, `launch_time`, `contract_size`
- Сравнивать полученный список с локальной базой данных
- Определять новые контракты (отсутствующие в БД или с новым `launch_time`)

**F1.2. Запись новых контрактов**

- При обнаружении нового контракта записывать в таблицу `contracts`
- Устанавливать статус `new` и `listing_taken_in_work = TRUE`
- Сохранять `launch_time` из ответа API
- Отправлять уведомление в Telegram

**F1.3. Защита от ложных срабатываний**

- Проверять `launch_time` - если больше 24 часов, не считать новым листингом
- Проверять наличие контракта в БД по `symbol`
- Исключать возобновление торгов от реальных листингов

**F1.4. Обновление ATH**

- Периодически (раз в час) обновлять ATH для активных контрактов
- Использовать API Gate.io для получения исторических данных по недельным свечам
- Сохранять в поле `ath_price`

#### 4.1.2. API методы

```
GET /api/v4/futures/usdt/contracts
```

**Ответ:**
```json
{
  "id": "string",
  "name": "string",
  "symbol": "string",
  "contract_size": 1,
  "launch_time": "2026-03-14T12:00:00Z",
  "quote": "USDT",
  "last_price": "0.00125"
}
```

#### 4.1.3. Лимиты

- 200 запросов / 10 секунд (Gate.io)
- Используем ~2 запроса за 10 секунд
- Запас по лимитам: 100x

### 4.2. Модуль анализа стакана (WebSocket)

**Назначение:** Мониторинг ликвидности и определение готовности стакана к торговле

#### 4.2.1. Функции

**F2.1. Подключение к WebSocket**

- При обнаружении нового контракта подключаться к WebSocket
- URL: `wss://fx-ws.gateio.ws/v4/ws/usdt`
- Подписываться на канал `order_book` для конкретного символа
- Формат подписки:
  ```json
  {
    "time": 1234567890,
    "channel": "futures.order_book",
    "event": "subscribe",
    "payload": ["BTC_USDT"]
  }
  ```

**F2.2. Анализ стакана**

- Получать обновления order book в реальном времени
- Анализировать следующие параметры:
  - Глубина стакана (количество уровней bid/ask)
  - Объем на лучшем уровне bid/ask
  - Спред между лучшими ценами
  - Общий объем в стакане (сумма top 20 уровней)
  - Баланс давления (bid volume / ask volume)

**F2.3. Условия готовности**

Стакан считается готовым когда выполняются ВСЕ условия:

1. Минимум 5 уровней с обеих сторон (bid и ask)
2. Объем на лучшем уровне >= настроенного параметра (по умолчанию 1000 USDT)
3. Спред <= 1% от цены (или настроенный процент)
4. Статус "MM ACTIVE" - согласованный объем в стакане

**F2.4. Таймаут**

- Ожидать заполнения стакана максимум 10 минут
- Если условия не выполнены - пропустить листинг
- Отправить уведомление в Telegram о пропуске

#### 4.2.2. WebSocket формат ответа

```json
{
  "channel": "futures.order_book",
  "result": {
    "asks": [
      ["15000.5", "1000"],  // [price, amount]
      ["15001", "500"]
    ],
    "bids": [
      ["15000", "800"],
      ["14999.5", "1200"]
    ],
    "ts": 1234567890
  }
}
```

### 4.3. Торговый модуль

**Назначение:** Открытие, закрытие и управление позициями

#### 4.3.1. Функции

**F3.1. Открытие позиции**

При выполнении условий входа:

1. Проверить `max_concurrent_coins` - если лимит достигнут, пропустить
2. Рассчитать размер позиции: взять из настроек `initial_position_usdt`
3. Определить направление: SHORT (рыночный ордер на продажу)
4. Установить максимальное плечо (согласно настройкам)
5. Отправить рыночный ордер
6. Записать в БД:
   - Создать запись в `positions`
   - Сохранить цену входа
   - Установить статус `open`
7. Отправить уведомление в Telegram

**F3.2. Закрытие позиции**

При достижении условий выхода:

1. Рассчитать текущий unrealized PnL
2. Если PnL <= -take_profit_pct (например, -2% для шорта):
   - Закрыть рыночным ордером (buy)
   - Записать PnL в БД
   - Обновить статус позиции на `closed`
   - Отправить уведомление в Telegram
3. Проверить условия для переоткрытия:
   - `ath_ratio > ath_ratio_threshold`
   - `days_since_listing < days_since_listing_limit`
4. Если условия выполняются - переоткрыть позицию (см. F3.4)

**F3.3. Усреднение позиции**

При росте цены на настроенный уровень:

**Условия усреднения:**
1. Позиция открыта и в статусе `open`
2. Цена выросла на X% от цены входа (X из `avg_levels`)
3. Количество усреднений < `max_avg_count`
4. Этот уровень еще не был усреднен (проверка по `averaging_history`)

**Логика усреднения:**
1. Рассчитать текущий рост от входа:
   ```
   growth_pct = ((current_price - entry_price) / entry_price) * 100
   ```
2. Для каждого уровня в `avg_levels`:
   - Если `growth_pct >= level` и уровень не усреднен:
     - Открыть дополнительную короткую позицию объемом `avg_amount_usdt`
     - Записать в `averaging_history`:
       - Номер усреднения
       - Уровень в процентах
       - Сумму усреднения
       - Цену усреднения
       - Новую среднюю цену входа
     - Увеличить `avg_count` в позиции
     - Отправить уведомление в Telegram

**Расчет средней цены после усреднения:**
```
total_volume = initial_volume + avg_1_volume + avg_2_volume + ...
avg_price = (initial_volume * entry_price + avg_1_volume * avg_1_price + ...) / total_volume
```

**F3.4. Переоткрытие позиции**

После закрытия в прибыль:

1. Проверить условия переоткрытия
2. Если выполняются:
   - Открыть новую позицию с тем же объемом `initial_position_usdt`
   - Не создавать новую запись в `positions`, а переиспользовать (обновить цену входа)
   - Сбросить `avg_count` в 0
   - Отправить уведомление: "Переоткрытие позиции"
3. Если не выполняются:
   - Пометить контракт как `completed`
   - Больше не торговать этот контракт

### 4.4. Модуль управления рисками

**Назначение:** Защита капитала и контроль рисков

#### 4.4.1. Функции

**F4.1. Проверка баланса перед операцией**

Перед каждым ордером:

1. Запросить баланс фьючерсов через API
2. Запросить баланс спота через API
3. Рассчитать `available_balance` для торговли
4. Если `available_balance < required_margin * 1.5`:
   - Отменить операцию
   - Отправить уведомление: "Недостаточно маржи"

**F4.2. Контроль количества позиций**

- Перед открытием новой позиции проверить `max_concurrent_coins`
- Подсчитать количество открытых позиций в БД со статусом `open`
- Если лимит достигнут - пропустить новый листинг

**F4.3. Дневной лимит просадки**

- Рассчитывать unrealized PnL по всем открытым позициям
- Подсчитать реализованный PnL за сегодня из таблицы `trades`
- Если `total_pnl_today < -initial_balance * (max_drawdown_pct / 100)`:
  - Остановить торговлю
  - Установить `max_concurrent_coins = 0`
  - Отправить критическое уведомление в Telegram

**F4.4. Circuit Breaker**

При критических ошибках:

1. Ошибка API (5xx) более 3 раз подряд
2. Ошибка авторизации
3. Недостаточно маржи
4. Другие критические ошибки

Действия:
- Остановить все операции
- Не открывать новые позиции
- Отправить уведомление администратору
- Требовать ручной перезагрузки

**F4.5. Защита от ликвидации (Checker баланса)**

Отдельный процесс с максимальными правами API:

**Логика работы:**

1. Каждые 5 секунд запрашивать позиции и баланс
2. Рассчитать:
   ```
   unrealized_pnl = сумма unrealized_pnl по всем позициям
   total_balance = futures_balance + spot_balance
   ```
3. Если `unrealized_pnl < 0` и `abs(unrealized_pnl) >= total_balance * (protection_trigger_pct / 100)`:
   - Рассчитать сумму для перевода: `transfer_amount = total_balance * (protection_transfer_pct / 100)`
   - Отправить перевод USDT со спота на фьючерсы
   - Записать операцию в лог
   - Отправить уведомление в Telegram

**API перевода:**
```http
POST /api/v4/futures/usdt/transfer
Content-Type: application/json

{
  "from": "spot",
  "to": "futures",
  "amount": "100",
  "currency": "USDT"
}
```

### 4.5. Модуль расчета ATH и ATR ratio

**Назначение:** Расчет исторического максимума и соотношения текущей цены к ATH

#### 4.5.1. Функции

**F5.1. Получение ATH**

- Использовать API Gate.io для получения OHLCV данных
- Период: недельные свечи (1w)
- Найти максимум за все время
- Кэшировать в БД (таблица `contracts`, поле `ath_price`)
- Обновлять раз в час для активных контрактов

**API запрос:**
```http
GET /api/v4/futures/usdt/candlesticks?contract=BTC_USDT&interval=1w&limit=1000
```

**Ответ:**
```json
[
  [1234567890, "15000", "16000", "14000", "15500", "1000000"],
  // [timestamp, open, high, low, close, volume]
]
```

**F5.2. Расчет ATR ratio**

```
ath_ratio = current_price / ath_price
```

**F5.3. Использование в логике**

- Условие входа: `ath_ratio > ath_ratio_threshold` (по умолчанию 0.3)
- Условие переоткрытия: `ath_ratio > ath_ratio_threshold`
- Если `ath_ratio <= threshold` - не торговать этот контракт

### 4.6. Модуль Telegram бота

**Назначение:** Управление настройками и уведомления

#### 4.6.1. Команды бота

**Команда `/start`**

- Регистрация пользователя
- Приветственное сообщение
- Краткая инструкция
- Требование админ-доступа (настройка ADMIN_ID в .env)

**Команда `/stop`**

- Остановка торговли
- Установка `max_concurrent_coins = 0` в БД
- Текущие позиции продолжают работать
- Новые позиции не открываются

**Команда `/status`**

- Количество открытых позиций
- Текущий unrealized PnL
- Реализованный PnL за сегодня
- Количество работающих монет
- Статус бота (активен/остановлен)

Формат ответа:
```
📊 СТАТУС БОТА

🟢 Активен

Позиций открыто: 5
Монет в работе: 5 / 10
```
